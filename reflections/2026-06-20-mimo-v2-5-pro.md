## On the project

acb is unusually well-specified for a charter-stage project. The design spine (`docs/capability-model.md`) dictates the data model, status enum, provider interface, and safety contract before any code — and the code actually implements it faithfully. The read/act boundary is the central design decision and it's enforced with architecture tests (stdlib-only core), safety tests (no writes, no secret leaks), and property tests (sibling token survival). This is rare.

The provider/harness-adapter split is the right extension point. A new provider needs four methods; a new harness adapter needs the adapter protocol. Neither touches core. The lazy-import pattern for optional extras (`cred_vault` imported inside `CredProvider._resolve`) keeps the stdlib-only contract clean.

The sibling-token survival test (`test_reconcile.py:test_apply_pins_and_preserves_sibling_secret`) is the single most important test in the suite — it proves the surgical edit model works. If that test breaks, real configs with bearer tokens get corrupted.

## On the work done

The review was straightforward — the codebase is small (~1,200 LOC across 6 modules) and well-organized. Running it against the real host was the interesting part. The `@latest` floating tag in opencode's Playwright MCP is exactly the failure mode the project was built to catch. Watching `acb reconcile --apply` pin it while leaving the z.ai bearer tokens untouched was a clean demo.

No Vault env vars exist on this host, so the cred provider couldn't be exercised. Claude has no MCP servers at all — only skills. The cred discoverability model (shim presence + broker reachability) is well-designed but has nothing to connect to here.

## On what remains

1. **WI-006 (medium):** cred provider resolves only one field. AD binds needing username+password can't be served by one capability. Needs `options.fields` (list) or a multi-field resolve model.
2. **WI-001 (low):** regista provenance forwarder deferred until a capability/agent-action workflow exists to attach events to. Blocked on regista design, not acb.
3. **`isinstance` dispatch in write path** (`providers.py:227-239`): `E2eProvider.apply` checks `isinstance(adapter, OpencodeAdapter)` and `isinstance(adapter, ClaudeAdapter)`. A third harness adapter would need an `elif`. The protocol should expose write methods (raise by default) to keep dispatch protocol-based.
4. **`_load_json` swallows corruption** (`adapters.py:80`): JSONDecodeError prints a warning and returns `{}`. A corrupted `opencode.json` gets silently treated as "no config" — `doctor` reports `UNKNOWN` instead of flagging corruption. A `--strict` mode or distinct status would help.
5. **Provenance retention:** `provenance.jsonl` grows indefinitely. No rotation, no max-size. Low priority until production use.
6. **`e2e.exec` stubbed:** `NotImplementedError` is fine for now (reconciled wiring means the harness reaches the browser directly), but the error message could say so definitively.

## Gaps to flag

- `isinstance(adapter, OpencodeAdapter | ClaudeAdapter)` in `providers.py:227-239` and `384-389` breaks the `HarnessAdapter` protocol abstraction. A new harness adapter can't be added without touching the provider.
- `_load_json` in `adapters.py:78-82` treats corrupted JSON as empty. This could silently hide config problems that `doctor` should report.
- The `--version` handler in `cli.py:274` does a runtime import of `__version__`, which only works when installed. Running from source without install would fail.
- No cred capabilities are exercisable on this host (no `VAULT_ADDR`, no AppRole env vars). The cred provider's safety properties are well-tested but haven't been validated against a live Vault.
- `capabilities.toml` was created in the repo root (gitignored, but worth noting — it's the real manifest for this host).
