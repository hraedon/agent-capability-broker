## On the work done

The host work was the substantive part, and the most useful thing I did was **not trust the request's premise**. "mvmcitest01 has 3.14.6 installed" was wrong in two ways (per-user, and 3.14.5 not 3.14.6). A survey under `svc-da` found only 3.12.8 and a 3.14.5 trapped in a backup venv. Removing 3.12 on that premise would have bricked the host's Python. I stopped, surfaced the discrepancy, and only after the user pointed me at `cw-admin` did the per-user 3.14.5 appear. The gating instinct (don't do a destructive system change on a false premise) was the highest-value behavior this session.

The SSH approach is worth recording: key auth was set up for `cw-admin` but the brokered `cred:svc-da` is a *different* account, so I drove password SSH via a stdlib `pty` helper invoked under `acb exec cred:svc-da` — inject-don't-surface held, the password never reached stdout or my context. PowerShell-over-SSH quoting is genuinely hostile; I abandoned inline multi-line commands for base64 `-EncodedCommand`, and had to defeat Windows PowerShell 5's `Set-Content -Encoding UTF8` BOM (which breaks `tomllib` at line 1 col 1) with a BOM-less `[IO.File]::WriteAllText`. Two smoke-test "failures" early on were both my test-harness artifacts (Store `py.exe` stub + BOM), not acb bugs — worth remembering before crying wolf on a compatibility report.

The 3.12 removal itself was messy: the MSI chain uninstalled Core/StdLib/Executables cleanly but six feature MSIs returned `1603` (they couldn't reconcile the already-removed core), and the cached bootstrapper under `svc-da`'s profile returned `1601` (ACL-protected cross-user). I finished with manual cleanup — delete the `Python312` tree, strip `HKLM PythonCore\3.12`, delete the six Uninstall registry keys, and edit the Machine PATH (backed up first). That's not a textbook uninstall; it's an admin making the end state correct. The re-validation is what certifies it: 3.14 is the sole Python, `py -0p` shows only 3.14, and acb is green.
