# Session Reflection — 2026-07-03

**Work summary:** Implemented Plan 005 (Suite cohesion: capability provisioning) across all three phases: suite config adoption (`$AGENT_SUITE_CONFIG` manifest/vault-env resolution + doctor JSON suite shape), `acb install-harness` bootstrap command, cross-platform shim rendering, and suite secret backend support. Adversarial review caught four issues (per-plane probing defeated by process env merge, XDG inconsistency, provenance gap for manual actions, unquoted Windows cmd pattern) — all fixed with tests. 26 new tests added (104 total, all green).

---

## On the project

acb has a clean, well-documented architecture. The design spine (`docs/capability-model.md`) is genuinely useful — it dictates the data model and the read/act boundary clearly, and the code follows it faithfully. The stdlib-only core constraint is well-enforced by the architecture test, though that test has a blind spot for relative imports (noted in the review). The provider/adapter pattern is elegant: providers stay harness-agnostic, adapters encapsulate format differences. The inject-don't-surface discipline is the project's most important safety property, and it holds.

The codebase is small enough (under 500 LOC in the core) that you can hold the whole model in your head, which is a sign of good factoring. The test suite is thorough on the happy paths and the key safety properties.

## On the work done

The suite config adoption (WI-1.1) was the most interconnected change — `suite_config_dir()` needed to be in `model.py` (stdlib-only core) but used by both `model.py` (manifest resolution) and `cred_vault.py` (vault env resolution). The XDG bug the reviewer caught was a good example of why consistency matters: `model.py` respected `XDG_CONFIG_HOME` but `cred_vault.py` didn't, and the test suite didn't catch it because the host doesn't have a custom XDG.

The `install-harness` command (WI-2.1) was straightforward — it's essentially `reconcile --apply` filtered to one harness, followed by a focused `doctor`. The provenance gap the reviewer found was a real consistency issue: `reconcile` emits provenance for manual actions (via `provider.apply()` returning a "skipped" result), but `install-harness` was `continue`-ing before reaching the emit call. The fix was to route all actions through `provider.apply()`.

The cross-platform shim rendering (WI-3.1) was simple but the reviewer's catch about quoting the Windows cmd value was valuable — `set "ACB_VAULT_ENV=path"&& ...` is safer than `set ACB_VAULT_ENV=path&& ...` because it handles special characters in paths.

The MEDIUM-1 fix (per-plane probing) was the most important: `_vault_env()` with an explicit `env_path` now skips the process env merge entirely. This is correct — when you're probing a specific access plane, you should use that plane's credentials, not whatever happens to be in the shell. The existing tests didn't catch this because they monkeypatched `cred_vault.reachable` and never exercised the real `_vault_env` merge.

## On what remains

- **The architecture test blind spot (MEDIUM-4 from review).** The `_imports` function in `test_arch.py` skips relative imports (`node.level == 0` filter), so `from . import cred_vault` at module level would be invisible. The current code is correct (lazy imports inside function bodies), but this is defense-in-depth that should be added.
- **Inline comment stripping in `_parse_env_text` (LOW-3 from review).** `VAULT_ADDR=https://vault # comment` sets the value to `https://vault # comment`. Minor but confusing.
- **`install-harness` doesn't update existing shims when the manifest changes (LOW-2 from review).** This is by design (no-clobber for hand-edited shims), but the docs could mention it.
- **`$AGENT_SUITE_CONFIG` pointing at a non-existent path silently returns that path (LOW-1 from review).** No warning is emitted. Benign but surprising.
- **WI-3.1 Windows testing.** The shim rendering includes Windows patterns, but no actual Windows CI exists. The `subprocess.run` with `env=` injection works cross-platform, but the rendered `set "KEY=val"&&` cmd pattern is untested on actual Windows.
- **Suite deployment validation.** Plan 005 says "a dual-harness validation confirms the cohesion changes don't regress an existing opencode config." This hasn't been done against a live suite deployment.

## Gaps to flag

- **`test_arch.py:35`**: The `node.level == 0` filter means relative imports are never checked. A core module could `from . import cred_vault` at module level and the test would pass. Add a check for relative imports of known optional-extra modules.
- **`cred_vault.py:102-110`**: The `_vault_env` fix is correct but the test (`test_vault_env_explicit_path_not_overridden_by_process_env`) only covers the `_vault_env` function directly. No end-to-end test verifies that `doctor` actually probes per-plane when `VAULT_ADDR` is in the shell env. The existing `test_inspect_probes_per_capability_vault_env_plane` monkeypatches `reachable` so it doesn't exercise the real merge path.
- **`cli.py:330-350`**: `install-harness` has no try/except around `provider.apply()`. If apply raises (e.g., `_create_file` raises `FileExistsError` on a race), the command crashes with an unhandled traceback. `reconcile` has the same gap.
- **No test for `install-harness` with multiple capabilities.** All tests use a single `cred:svc-bot` capability. Multiple capabilities might expose ordering or interaction issues.
- **`providers.py:330`**: The Windows cmd pattern `set "ACB_VAULT_ENV={ve}"&& ...` has no space between the closing quote and `&&`. This is correct for cmd (spaces after `set` are significant), but it's non-obvious and could confuse someone reading the shim.
