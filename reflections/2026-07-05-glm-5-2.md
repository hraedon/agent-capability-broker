## On the project

The repo is in a healthy, mostly-settled state. Plans 001–005 are all landed,
116 tests green, ruff/mypy clean, the CLI exposes exactly the five verbs the
docs now claim. The interesting friction is the one this session surfaced:
status footers in long-lived docs drift silently from "first deliverable"
language to reality. There's no mechanism that flags this — it relies on
whoever-next-touches noticing, which is exactly the chore I just did. A
cheap mitigation would be a tiny test that greps the Status sections for
"first deliverable"/"Proposed" and fails when the verb count in `acb --help`
exceeds what the footer claims, but that's probably over-engineering for a
repo this size.

## On the work done

Genuinely trivial change — three .md edits, no logic. What I'd flag: I
initially wrote the wrong landing date (copied the *proposal* date 07-02
instead of checking the commit timestamp, which is 07-03). The adversarial
reviewer caught it. Lesson reinforced: when a header says "Proposed
<date>", the landing date is a *different* fact and must be read from
`git log -1 --format=%cI <sha>`, not transcribed from the line above. The
reviewer also flagged a NIT about the Plan 005 title using a colon where
Plans 001–004 use an em-dash only — I left it, since the title is the
original author's stylistic choice and not part of the "stale header" task
scope. Worth knowing it exists.
