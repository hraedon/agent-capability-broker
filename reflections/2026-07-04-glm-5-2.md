## On the work done

### Gaps from prior reflection (all fixed)

1. **Architecture test blind spot (MEDIUM-4):** The original fix removed `node.level == 0` but still missed `from . import cred_vault` because `node.module` is `None` for that import form. The adversarial review caught this — the test gave false assurance about the exact pattern used in `providers.py`. Fixed by checking only `tree.body` (module-level statements) and handling `node.module=None` by capturing alias names. Now both `from .cred_vault import X` and `from . import cred_vault` at module level are caught, while function-level lazy imports (the correct pattern) are ignored.

2. **Inline comment stripping (LOW-3 → HIGH-2):** The initial fix was naive — it stripped comments before quote stripping, corrupting quoted values like `"abc # def"`. The adversarial review caught this as HIGH. Fixed by rewriting the parser to handle quoted values first: if the value starts with a quote, the closing quote ends the value and anything after is a comment. This correctly handles `"abc # def" # real comment`.

3. **try/except around provider.apply():** Added to both `reconcile` and `install-harness`. Catches `(OSError, KeyError)` — the exceptions adapters actually raise. `FileExistsError` was removed from the tuple (it's a subclass of `OSError`, so redundant).

4. **No test for multiple capabilities:** Added `test_install_harness_multiple_capabilities` — two creds provisioned in one run.

5. **No test for reconcile apply errors:** Added `test_reconcile_apply_error_is_handled` — patches `E2eProvider.apply` to raise `OSError`, asserts `[FAILED]` output and non-zero exit code.

### New bug discovered during Windows validation

**argparse REMAINDER bug:** When running `acb exec cred:svc-bot -m manifest.toml -- cmd`, `argparse.REMAINDER` captures `-m manifest.toml` as part of the command argv instead of parsing it as the `--manifest` option. This caused `subprocess.run(["-m", "manifest.toml", ...], env=child_env)` → `FileNotFoundError: [WinError 2]` on Windows.

This was a real cross-platform bug — it would have affected Linux too, but the existing tests always put `-m` before the capability (`["exec", "-m", manifest, "cred:test", "--", *child]`), so it was never caught. The Windows validation exposed it because PowerShell naturally puts the flag after the positional.

Fixed by manually extracting `-m`/`--manifest` from `args.argv` before processing. The `--` separator stops extraction, so child commands that use `-m` (after `--`) are preserved.
