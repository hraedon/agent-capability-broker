# Session Reflection — 2026-06-20 (session 3)

**Work summary:** Replicated the cert-watch Vault roles for both harnesses (opencode + claude, mirroring ci/e2e/totp-mcp), wired per-access-plane `.env` files, and added `options.vault_env` so each capability's shim points at the right AppRole file. The homelab and cert-watch access planes are now fully separated — a compromised homelab role yields nothing from cert-watch and vice versa. All cert-watch capabilities validated end-to-end against live Vault.

---

## On the project

The per-capability `vault_env` design turned out to be the right abstraction. When I first built the `.env` fallback, each harness had one vault file. Adding cert-watch forced the question: how does a capability select which AppRole to use? The answer (`options.vault_env`, a bare filename resolved beside the harness config) is clean — it's declarative in the manifest, the shim embeds the right path, and `cred_vault` doesn't need to know about it (it just reads `ACB_VAULT_ENV`). The capability manifest is now the single source of truth for which access plane each credential belongs to.

The access-plane separation mirrors what the cert-watch Vault policies already intended — the e2e-runner policy comment says "Keeping them split means a leak of one credential plane never yields the other." acb now extends that principle to the harness layer: each harness has a homelab role AND a cert-watch role, independently revocable.

## On the work done

**What went well:** The cert-watch role replication was mechanical — read the source policies, write mirror copies with the harness prefix, create AppRoles, generate creds. The least-privilege verification (totp-mcp denied passwords, e2e can read them) confirmed the policies are correctly scoped. The per-capability `vault_env` code change was small (one helper function + one call-site update + one test).

**What was awkward:** I initially ran `acb exec` without the `ACB_VAULT_ENV` prefix and got "VAULT_ADDR is not set." The shim tells the agent to prefix the command, but I forgot. An agent reading the shim would do it right; a human at the terminal has to remember. This is fine — the shim is the discovery artifact — but it's a UX rough edge for manual testing.

**What I'm confident in:** The per-capability routing works exactly as designed. `cred:svc-da` → `vault.env` (homelab role), `cred:ldap-bind` → `cert-watch.env` (cert-watch role). Verified by the shim content and by the live exec validation. 71 tests green.

**What I'd want a second pair of eyes on:** The `_vault_env_path` override resolves a bare filename against the adapter's vault dir. A capability could set `vault_env = "../../../etc/passwd"` and the shim would embed that path — but it's only a path in a markdown file the agent reads; acb doesn't execute it. The `ACB_VAULT_ENV` env var is set by the shell when the agent runs the command. So the path traversal is a theoretical concern only (the path appears in the shim text, but acb resolves it via `Path.read_text`). Still, a path sanitization check wouldn't hurt.

## On what remains

1. **WI-008 (medium, filed this session):** doctor can't validate per-capability AppRoles in one run — it uses a single `ACB_VAULT_ENV` from the shell for all reachability checks. In a multi-plane setup this means doctor over-claims PRESENT_OK. The fix is to have doctor resolve the per-capability vault_env path when probing each capability.
2. **WI-001 (low, deferred):** regista provenance forwarder — blocked on regista workflow design.
3. **TOTP code brokering:** the `cert-watch-totp-mcp` roles exist but acb doesn't have a provider for TOTP code reads (the `totp/code/` path is a different Vault secrets engine, not KV). This would be a new capability type if agents need to broker MFA second factors. Currently out of scope — the e2e-runner role reads TOTP codes directly, which works for the CI test process.
4. **The `isinstance` dispatch in providers.py** (pre-existing) — still there, still a future-harness concern.

## Gaps to flag

- **doctor over-claims in multi-plane setups (WI-008).** `cred_vault.reachable()` reads `ACB_VAULT_ENV` from the shell, not per-capability. A doctor run with the homelab env reports PRESENT_OK for cert-watch capabilities even if the cert-watch AppRole is misconfigured — the token self-lookup succeeds regardless. Filed as WI-008.
- **`_vault_env_path` doesn't sanitize the override.** `providers.py:283` resolves `options.vault_env` as `base.parent / override`. A path-traversal override (`../../...`) would appear in the rendered shim text. Harmless in practice (acb reads it as a file path, and the manifest author controls it), but a `Path.name`-based sanity check would be defense-in-depth.
- **No test covers the cert-watch access-plane separation end-to-end in CI.** The live validation confirmed it works, but the unit tests mock `cred_vault.resolve` — they don't prove that a capability with `vault_env = "cert-watch.env"` would actually authenticate via the cert-watch role at exec time. The new `test_plan_renders_per_capability_vault_env` proves the shim embeds the right path, which is the mechanism, but not the full auth flow.
- **`cert-watch-setup` role was deliberately not replicated.** It's a bootstrap/maintenance role (creates policies, enables engines). If an agent ever needs to stand up cert-watch wiring, that role would need a harness mirror. Currently out of scope for CI tests.
