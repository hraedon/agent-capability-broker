# Session Reflection — 2026-06-20

**Work summary:** Implemented WI-006 (multi-field Vault resolve) and per-harness AppRole `.env` auth so both Claude and opencode can broker AD credentials for Windows CI tests. Created separate opencode AppRoles in Vault, wired `.env` files for each harness, and validated end-to-end against live Vault. Two adversarial reviewers caught real issues (all-fields footgun, `.env` parser gaps, hvac exception leaks); all were fixed before commit. 70 tests green.

---

## On the project

This session moved acb from "promising charter-stage tool" to "working credential broker for the lab." The design spine's inject-don't-surface contract held up perfectly under real use: a child process receives `USERNAME`/`PASSWORD` from Vault and the secret never appears in stdout, stderr, or provenance. The per-harness AppRole separation (`ACB_VAULT_ENV` pointing each harness at its own `.env`) is clean — Claude and opencode hold distinct least-privilege roles, revocable independently.

The fail-closed default for field selection was the right call, forced by the adversarial review. My original "default to all fields" was a footgun: a Vault secret isn't curated for injection, and defaulting to "dump everything" risks over-exposing side-channel material. The reviewers earned their keep on that one.

## On the work done

**What went well:** The end-to-end validation was clean. `acb exec cred:svc-da` against the real Vault injected `username`, `upn`, and `password` in one shot — exactly the AD bind case that was previously impossible. The permission-denied case (`cred:ldap-bind`, which the opencode role intentionally can't read) now fails with a one-line error instead of a traceback, thanks to the hvac exception wrapping.

**What the adversarial review caught that I missed:**
1. The all-fields default was a real footgun (both reviewers, independently).
2. `export VAULT_...` lines in the `.env` file would silently break auth (both).
3. The `test_cred_vault_multi.py` module-level `import hvac` would break no-`[cred]`-extra CI runs — the `importorskip` fix was important.
4. The missing-fields error message leaked the names of the secret's other fields — removed the `available:` parenthetical.
5. TOCTOU in `_load_env_file` (is_file → read_text race).

**What I'm confident in:** The injection machinery, the fail-closed validation, the `.env` parser robustness, and the test coverage of the safety properties (no secret leakage in any error path).

**What I'd want a second pair of eyes on:** The `_vault_env()` merge semantics (process env overrides `.env` file) — I believe this is correct but it's the kind of thing where a subtle ordering bug could surprise. Also, the `cred:ldap-bind` capability reports PRESENT_OK in doctor but fails at exec (see Gaps).

## On what remains

1. **WI-001 (low, deferred):** regista provenance forwarder — blocked on a capability/agent-action workflow existing in regista. Not acb's blocker.
2. **`cred:ldap-bind` needs a cert-watch-scoped role** — the opencode/claude AppRoles only cover `homelab/ad/*`. For cert-watch E2E tests, a separate AppRole with `cert-watch` policy is needed. This is a Vault setup task, not a code change.
3. **Global install path:** `acb` is at `~/.local/bin/acb` via `uv tool install`. This works but isn't documented anywhere agents would discover it. The shims reference `acb` (bare), which relies on PATH. Worth a note in AGENTS.md or a setup script.
4. **The `isinstance` dispatch in providers.py** (pre-existing, noted last session) — still there, still a future-harness concern.

## Gaps to flag

- **Doctor over-claims for path-unauthorized capabilities.** `cred:ldap-bind` reports `PRESENT_OK` (shim present, broker reachable via token self-lookup) but `acb exec` fails with permission denied. This is designed behavior (spine §4: reachability ≠ authorization, self-lookup is not a secret read), but the UX is misleading. A future `doctor --deep` that attempts a no-op read could catch this, but that blurs the read/act boundary. Leave as-is but document it.
- **The `_vault_env()` merge** in `cred_vault.py:48-52` picks up ALL `VAULT*` process env vars, not just known ones. A typo like `VAULT_ADRESS` would be silently ignored by hvac but present in the dict. Harmless but noisy — an allowlist would be tighter.
- **`vault.env` permissions are not checked.** `adapters.py:vault_env_path` returns the path without verifying mode 0600. A world-readable `vault.env` holding AppRole material is a real risk. A defense-in-depth warning on the act path would help.
- **No test covers `acb exec` through the full CLI with the vault source** — the new `test_multi_field_exec_injects_each_and_stays_silent` mocks `cred_vault.resolve`, which proves the injection path but not the CLI plumbing for the vault source end-to-end. The live validation covered this, but it's not pinned in CI.
